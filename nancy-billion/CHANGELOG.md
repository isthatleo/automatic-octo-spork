## 2026-07-12 — Repo cleanup + orchestration/LLM revamp (Phase 1)

Kept only docs that reflect real state or intent: `README.md`, `TODO.md`, `SOVEREIGN_CAPABILITIES_PLAN.md`, `IMPLEMENTATION_PLAN_INTELLIGENT_OS.md`, `NANCY_STARTUP_GREETINGS.md`, `NANCY_PERSONALIZED_GREETING.md`, `LLM_SETUP_GUIDE.md`.
- Deleted dead code: `backend/main.py`, `backend/main_backup.py` (both broken/superseded — `backend/main_new.py` is the real entry point), 12 orphaned `tools_*.py` iteration artifacts (only `backend/tools.py` is live), `backend/voice/streaming.py` (unused duplicate of `wake_word.py`), `backend/services/{context-bridge,environmental-awareness,proactive-assistant}.py` (broken hyphenated-filename modules, never importable).
- Deleted one-off root migration/patch scripts (`fix_*.py`, `add_agent.py`, `add_learning_division.py`, `learning_*.py/.txt`, `rebuild_tools.py`, `sovereign_control_*.py`) — verified none were imported by the live `tools.py`.
- **Correction during cleanup:** `agent_executor.py`, `agents/base.py`, `agents/chief_autonomy.py`, `agents/knowledge_synthesis.py`, `agents/strategic_planning.py`, and `agents/real_compute.py` were initially flagged as dead but are actually load-bearing (imported via `agents/__init__.py` and by 16+ specialized agents) — kept.
- Fixed the silently-broken orchestration path: `main_new.py`'s `/chat` endpoint imported `orchestration.integration.run_nancy_hierarchy`, whose relative imports (`from ..llm import ...`) failed at runtime — every chat request silently fell back to a bare LLM call despite status docs claiming multi-agent orchestration was live. Replaced with real routing through `agent_service` (the working 29-agent runtime already exercised by `/agents/run`/`/agents/auto`): chat text is now matched against the same keyword-routing table `auto_run` uses, and delegated to a real specialized agent on a match, falling back to the general LLM otherwise. Deleted the now-fully-orphaned `orchestration/` package.
- Rewrote `AnthropicLLM` (`backend/llm.py`) to use the official `anthropic` Python SDK instead of hand-rolled `aiohttp` calls; updated the default model from the retired `claude-3-opus-20240229` to `claude-opus-4-8`; added real token streaming via `generate_stream()`. Sampling params (`temperature`/`top_p`/`top_k`) are intentionally never sent to the API — Opus 4.7+ rejects them with a 400.
- **Correction:** `config/settings.py` was flagged in the plan as fully orphaned and slated for deletion — verified this is wrong. `agents/base.py` (live, load-bearing) imports `utils/logger.py`, which imports `config/settings.py` for logging configuration. Left both files in place; the LLM-provider config in `llm.py` (plain `os.getenv`) and the logging config in `config/settings.py` are two independent concerns, not a duplicate/broken pair.
- **Correction:** memory persistence audit was wrong — `memory/graph.py`'s `MemoryGraph` already saves/loads to `data/memory_graph.json` on every write, it's just JSON not SQLite. Not broken; deprioritized the SQLite rewrite in favor of higher-value fixes.
- **Plan pivot (OANDA unavailable in Uganda):** `trading/forex_engine.py`'s `ForexDataAggregator.get_price()`/`get_historical()` previously returned hardcoded mock prices through endpoints that looked like live data — the clearest "fake presented as real" issue in the whole codebase. Replaced with real ECB reference rates via Frankfurter (frankfurter.app — free, key-less, no geographic account restriction). This is real data at daily resolution (not live tick/bid-ask), which is now stated explicitly in the module docstring rather than implied to be more than it is; bid/ask is a small synthetic spread clearly labeled as such around the real mid rate. Order execution remains unconnected — no broker integration exists yet, and shouldn't be added without deciding on a broker that actually serves the account holder's country.
- Security hardening on `main_new.py`: CORS restricted to explicit origins (was `["...", "*"]`, which also silently broke `allow_credentials=True` per the CORS spec); added an opt-in shared-secret auth check (`BACKEND_AUTH_TOKEN`) and a per-IP sliding-window rate limiter, both no-ops by default for local dev, applied to `/chat`, `/agents/run`, `/agents/auto`, `/trading/*`, and the `/ws` websocket.

## 2026-07-12 — Phase 2: honesty audit of the 9 "advanced enhancement" agents

All 9 agents from TODO.md's ask (`artificial_consciousness_core`, `recursive_self_improvement_engine`, `ethical_governance_core`, `embodied_cognition_interface`, `temporal_prediction_engine`, `multi_agent_swarm_coordinator`, `quantum_reasoning_accelerator`, `neural_interface_agent`, `holographic_display_controller`, `environmental_control_nexus`) already existed (300-1400 lines each, registered, reachable via real `/agents/*` REST endpoints) — this pass audited each for honesty and safety rather than building from scratch:

- **`artificial_consciousness_core.py`** — real computation (Shannon entropy, KL-divergence, Bayesian updates, IIT-style Phi), but the docstrings/interpretation strings explicitly claimed actual sentience ("rich internal experience," "qualia-simulation" as a listed API capability, "Strong candidate for sophisticated conscious experience"). Rewrote all of it to honestly describe real introspection/metacognition metrics without claiming subjective experience, while keeping the underlying math unchanged.
- **`recursive_self_improvement_engine.py`** — confirmed it has zero filesystem/subprocess/exec access (cannot touch real code) and already had an automated `SafetyVerifier` gate, but no human-in-the-loop step — a caller could chain propose→verify→apply autonomously. Added a required `human_approve` step (with an `approved_by` identifier) between automated verification and `apply`.
- **`ethical_governance_core.py`** — found and removed a circular fake-confidence bug: Gaussian noise was added "to simulate deliberation uncertainty," then confidence was derived from that same synthetic noise. Replaced with a real signal (confidence scales with how clear-cut the aggregate score is).
- **`neural_interface_agent.py`, `holographic_display_controller.py`, `environmental_control_nexus.py`** — all three advertised real hardware integration (EEG headset SDKs, light-field displays, smart-home hubs) in their `capabilities`/`tools` fields and returned `"success"`/`"status": "success"` responses indistinguishable from real hardware actuation, with no Provider/Mock abstraction anywhere. Added explicit `mode: "simulated"` / `hardware_connected: false` flags, split "tools actually used" from "designed for future integration with," and relabeled response fields (e.g. `devices_affected` → `devices_that_would_be_affected`) so simulated results can never be mistaken for real hardware output.
- **`temporal_prediction_engine.py`** — the `model` field on each forecast (e.g. "LSTM", "Prophet") was chosen by `random.choice` and never actually influenced the computation (always linear-trend + Gaussian noise) — documented this explicitly rather than implying a specific algorithm ran.
- **`multi_agent_swarm_coordinator.py`** — confirmed it does NOT delegate to the system's other 28 real agents (purely internal simulated state); added an explicit docstring disclaimer pointing to `agent_service.py` as where real routing actually happens.
- **`quantum_reasoning_accelerator.py`** — re-verified already honest (explicit "classically simulated" disclaimers throughout); no changes needed.
- Added `mode`/`hardware_connected` fields to `base_specialized_agent.py`'s `get_info()` (used by `/agents/list`) so the frontend can badge simulated vs. production agents in the roster view, not just the per-agent detail view.
- Wired Claude specifically into coding/self-improvement/devops task routing: `select_llm_for_task()` now also matches those hints, and `main_new.py`'s `/chat` endpoint requests `effort="high"` (adaptive thinking) from `claude-opus-4-8` specifically for those task types.

## 2026-07-12 — CRITICAL FIX: backend/tools.py could not be imported at all

While running a final full-backend compile check (`python -m compileall`), discovered `backend/tools.py` — imported unconditionally at the top of `main_new.py` (`from tools import get_tools`) — was missing its entire header: no `logger`, no `add_tool()` function, no `get_tools()` function. Only ~630 lines of `add_tool(...)` call sites remained, plus null-byte corruption and mixed `\r\r\n`/`\r\n` line endings. **This meant `main_new.py` could not be imported, and the backend could not start, independent of any change made in this session** — confirmed pre-existing by checking `git show HEAD:nancy-billion/backend/tools.py` (same corruption already present in the last commit).

Fixed by: normalizing all line endings to `\n`, removing stray null bytes, fixing 13 instances of a double-escaped-quote corruption (`\\'` → `\'`, e.g. `'or \\'all\\'' ` broke string parsing), and reconstructing the missing `add_tool`/`get_tools`/`logger` header. The reconstruction is grounded in verified evidence, not guesswork: every `add_tool(id, description, execute, input_schema, output_schema)` call site uses the exact 5-positional-argument shape of `fury.types.create_tool()` (confirmed by reading `fury-main/src/fury/types.py`, the source for the `fury-sdk` dependency), so `add_tool` is a thin wrapper appending a `Tool` to a module list, and `get_tools()` returns that list. Also fixed the same UTF-16/null-byte corruption in the unrelated, unused `agents/test/test_agent.py` fixture. Full backend now passes `python -m compileall` with zero errors — verified end to end, not just for the files touched this session.
